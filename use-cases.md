## Use Case 1 — Σύνδεση Χρήστη & Πλοήγηση στο Σύστημα

| | |
|---|---|
| **Ονομασία Περίπτωσης Χρήσης** | Σύνδεση Χρήστη & Πλοήγηση στο Σύστημα |
| **Διεύθυνση (URL)** | `index.html` |
| **Δρώντες (Actors)** | Όλοι οι εσωτερικοί χρήστες (Operations, DPA/HSEQ, Technical, Accounting, Management, Πλοίαρχος) |
| **Σύντομη Περιγραφή** | Ο χρήστης συνδέεται στο ενοποιημένο ΠΣ της HIND Maritime εισάγοντας credentials και ρόλο. Μετά την επιτυχή σύνδεση, μεταβαίνει στο αρμόδιο module μέσω των navigation cards που εμφανίζονται στην αρχική. |
| **Προ-συνθήκες** | Ο χρήστης διαθέτει ενεργό λογαριασμό στο σύστημα και έγκυρα διαπιστευτήρια. |
| **Βασική Ροή** | 1. Ο χρήστης ανοίγει την αρχική σελίδα του ΠΣ.<br>2. Συμπληρώνει όνομα χρήστη, κωδικό και επιλέγει ρόλο.<br>3. Πατά «Είσοδος στο Σύστημα».<br>4. Το σύστημα ταυτοποιεί και ανακατευθύνει στο default dashboard του ρόλου.<br>5. Ο χρήστης επιλέγει το επιθυμητό module από τις navigation cards. |
| **Μετα-συνθήκες** | Ο χρήστης βρίσκεται στο επιλεγμένο module με δικαιώματα του ρόλου του. Δημιουργείται login event στο Audit Trail. |
| **Εναλλακτικές Ροές** | 3α. Λάθος credentials → εμφάνιση μηνύματος σφάλματος, επιστροφή στο βήμα 2.<br>3β. Ο χρήστης πατά «Ξέχασα τον κωδικό μου» → ξεκινά ροή password reset. |

## Use Case 2 — Άμεση Πρόσβαση στα Πιστοποιητικά & Εξαγωγή Vetting Folder

| | |
|---|---|
| **Ονομασία Περίπτωσης Χρήσης** | Προβολή Κατάστασης Στόλου & Εξαγωγή Compliance Report |
| **Διεύθυνση (URL)** | `fleet-status.html` |
| **Δρώντες (Actors)** | DPA / HSEQ Manager (Certification Executives), Operations Officer, Technical |
| **Σύντομη Περιγραφή** | Ο χρήστης βλέπει σε real-time την κατάσταση πιστοποιητικών όλου του στόλου με χρωματικές ενδείξεις (Valid / Expiring / Expired / Missing Data). Έχει τη δυνατότητα να εξάγει με ένα κλικ τον πλήρη ψηφιακό φάκελο Vetting για συγκεκριμένο πλοίο, έτοιμο για αποστολή σε επιθεωρητές. |
| **Προ-συνθήκες** | • Ο χρήστης είναι συνδεδεμένος.<br>• Έχει δικαίωμα πρόσβασης (τουλάχιστον Read-Only) στα πιστοποιητικά.<br>• Τα πιστοποιητικά έχουν καταχωρηθεί στο σύστημα με έγκυρες ημερομηνίες λήξης. |
| **Βασική Ροή** | 1. Ο χρήστης μεταβαίνει στο Fleet Status Dashboard.<br>2. Το σύστημα εμφανίζει KPIs (Compliant / Λήγουν / Expired) και πίνακα στόλου με badges κατάστασης.<br>3. Ο χρήστης μπορεί να φιλτράρει ανά πλοίο, τύπο πιστοποιητικού ή κατάσταση.<br>4. Επιλέγει ένα πλοίο και πατά «Export Vetting».<br>5. Το σύστημα συγκεντρώνει αυτόματα μόνο τα Valid πιστοποιητικά και παράγει ενιαίο αρχείο (.zip ή .pdf) σε <1 λεπτό.<br>6. Η εξαγωγή καταγράφεται στο Audit Trail με χρονοσήμανση και όνομα χρήστη. |
| **Μετα-συνθήκες** | Ο χρήστης διαθέτει ενημερωμένο Vetting Folder. Έχει ενημερωθεί για τυχόν λήξεις και έχει την δυνατότητα να προγραμματίσει ανανεώσεις. |
| **Εναλλακτικές Ροές** | 5α. Πιστοποιητικό Expired → αποκλείεται από το export, εμφανίζεται warning.<br>5β. Πλοίο με Missing Data → ειδοποίηση στο τμήμα Certification.<br>5γ. Τεχνικό σφάλμα στη δημιουργία αρχείου → αυτόματη ειδοποίηση στο IT. |

## Use Case 3 — Real-Time Παρακολούθηση Ταξιδιών (Voyage Dashboard)

| | |
|---|---|
| **Ονομασία Περίπτωσης Χρήσης** | Προβολή Στοιχείων Ταξιδιού σε Πραγματικό Χρόνο |
| **Διεύθυνση (URL)** | `voyage-dashboard.html` |
| **Δρώντες (Actors)** | Operations Officer, Management (read-only) |
| **Σύντομη Περιγραφή** | Παρέχει στο τμήμα Operations ένα real-time Voyage Dashboard που ενημερώνεται αυτόματα βάσει των Noon Reports (≤5 λεπτά). Εμφανίζει θέση, ταχύτητα, κατανάλωση καυσίμων, ETA και εκτιμώμενη κερδοφορία ανά ταξίδι. Διαθέτει ειδοποιήσεις αποκλίσεων από το πλάνο. |
| **Προ-συνθήκες** | • Ο χρήστης είναι συνδεδεμένος ως Operations.<br>• Υπάρχουν ενεργά ταξίδια στο σύστημα. |
| **Βασική Ροή** | 1. Ο χρήστης επιλέγει «Voyage Dashboard» από το navigation.<br>2. Το σύστημα εμφανίζει KPIs και πίνακα όλων των ενεργών ταξιδιών σε πραγματικό χρόνο.<br>3. Ο χρήστης εφαρμόζει φίλτρα (πλοίο, δρομολόγιο, περίοδο).<br>4. Σε περίπτωση σημαντικής απόκλισης (π.χ. καθυστέρηση ETA) εμφανίζεται automatic alert.<br>5. Ο χρήστης μπορεί να εξάγει δεδομένα σε Excel/PDF ή να εκδώσει νέες Voyage Instructions. |
| **Μετα-συνθήκες** | Το τμήμα Operations έχει σαφή εικόνα της κατάστασης του στόλου και μπορεί να λάβει ενημερωμένες αποφάσεις. |
| **Εναλλακτικές Ροές** | 4α. Καθυστέρηση Noon Report → ένδειξη «Αναμονή Noon Report» με την ώρα του τελευταίου ενημερωμένου στοιχείου.<br>4β. Αποσύνδεση πλοίου → εμφάνιση τελευταίας γνωστής θέσης με badge «Offline». |

## Use Case 4 — Υποβολή Noon Report από Πλοίαρχο

| | |
|---|---|
| **Ονομασία Περίπτωσης Χρήσης** | Ψηφιακή Υποβολή Ημερήσιας Αναφοράς (Noon Report) |
| **Διεύθυνση (URL)** | `noon-report.html` |
| **Δρώντες (Actors)** | Πλοίαρχος (Master), Operations (παραλήπτης) |
| **Σύντομη Περιγραφή** | Ο Πλοίαρχος υποβάλλει το ημερήσιο Noon Report μέσω τυποποιημένης ψηφιακής φόρμας από τη διεπαφή του πλοίου. Τα δεδομένα αναλύονται αυτόματα (Data Parsing) και ενημερώνουν την κεντρική βάση και το Voyage Dashboard, εξαλείφοντας τη χειροκίνητη πληκτρολόγηση από τη Γραμματεία. |
| **Προ-συνθήκες** | • Ο Πλοίαρχος είναι συνδεδεμένος στο σύστημα από τη διεπαφή του πλοίου.<br>• Υπάρχει ενεργό Voyage για το πλοίο. |
| **Βασική Ροή** | 1. Ο Πλοίαρχος επιλέγει «Υποβολή Noon Report».<br>2. Συμπληρώνει την τυποποιημένη φόρμα: θέση, καύσιμα, ταχύτητα, καιρός, φορτίο, ETA, παρατηρήσεις.<br>3. Πατά «Υποβολή Noon Report».<br>4. Το σύστημα επιβεβαιώνει παραλαβή με χρονοσήμανση εντός 1 λεπτού.<br>5. Τα δεδομένα αναλύονται αυτόματα (Data Parsing) και εισάγονται στην κεντρική βάση.<br>6. Εντός 5 λεπτών, το Voyage Dashboard του Operations ενημερώνεται με τα νέα στοιχεία. |
| **Μετα-συνθήκες** | Νέο Noon Report καταχωρείται στο σύστημα. Ενημερώνονται αυτόματα τα downstream modules (Operations dashboard, KPIs Διοίκησης, Accounting cost tracking). Δημιουργείται audit entry. |
| **Εναλλακτικές Ροές** | 3α. Ελλιπή υποχρεωτικά πεδία → το σύστημα επισημαίνει τα κενά και δεν επιτρέπει υποβολή.<br>3β. Απώλεια συνδεσιμότητας πλοίου → η αναφορά αποθηκεύεται τοπικά και αποστέλλεται αυτόματα μόλις αποκατασταθεί η σύνδεση («Αποθήκευση Πρόχειρου»). |

## Use Case 5 — Ψηφιακή Έγκριση Αιτήματος Προμήθειας από Διοίκηση

| | |
|---|---|
| **Ονομασία Περίπτωσης Χρήσης** | Έγκριση/Απόρριψη Αιτήματος Προμήθειας |
| **Διεύθυνση (URL)** | `approve-procurement.html` |
| **Δρώντες (Actors)** | Διοίκηση / CEO (αποφασίζων), Τμήμα Προμηθειών (παραλήπτης ειδοποίησης) |
| **Σύντομη Περιγραφή** | Η Διοίκηση εγκρίνει ή απορρίπτει ψηφιακά αιτήματα προμήθειας που έχουν υποβληθεί από τους Πλοιάρχους και έχουν προ-επεξεργαστεί από το Τμήμα Προμηθειών (συγκέντρωση προσφορών). Κάθε απόφαση καταγράφεται αυτόματα με χρονοσήμανση και όνομα στελέχους, αντικαθιστώντας πλήρως τη χειρόγραφη υπογραφή σε φυσικά έγγραφα. |
| **Προ-συνθήκες** | • Ο χρήστης είναι συνδεδεμένος ως Management/CEO με δικαιώματα έγκρισης.<br>• Υπάρχει αίτημα προμήθειας σε κατάσταση «Εκκρεμεί Έγκριση» με συγκεντρωμένες προσφορές. |
| **Βασική Ροή** | 1. Η Διοίκηση λαμβάνει in-app notification για εκκρεμές αίτημα.<br>2. Ανοίγει την οθόνη Approvals και βλέπει τη λίστα εκκρεμών αιτημάτων.<br>3. Επιλέγει συγκεκριμένο αίτημα και βλέπει αναλυτικά: περιγραφή, εκτιμώμενο κόστος, προσφορές, budget status, audit trail.<br>4. (Προαιρετικά) προσθέτει σχόλιο.<br>5. Επιλέγει «Έγκριση» ή «Απόρριψη».<br>6. Το σύστημα καταγράφει αυτόματα: χρονοσήμανση, όνομα στελέχους, απόφαση.<br>7. Εντός 1 λεπτού αποστέλλεται αυτόματη ειδοποίηση στο Τμήμα Προμηθειών.<br>8. Το αίτημα μεταβαίνει σε κατάσταση «Εγκεκριμένο» / «Απορριφθέν». |
| **Μετα-συνθήκες** | Το αίτημα έχει ολοκληρωθεί σε επίπεδο Διοίκησης. Έχει δημιουργηθεί πλήρες ψηφιακό ίχνος (Audit Trail). Η ροή εξακολουθεί downstream προς τον προμηθευτή. |
| **Εναλλακτικές Ροές** | 5α. Αίτημα εκτός ορίου εξουσιοδότησης → το σύστημα απαιτεί πρόσθετη έγκριση από ανώτερο στέλεχος (multi-level approval).<br>5β. «Παραπομπή σε Ανώτερο» → το αίτημα προωθείται στον CEO με σημείωμα.<br>5γ. Η Διοίκηση δεν απαντά εντός 24 ωρών → το σύστημα στέλνει αυτόματο reminder. |
